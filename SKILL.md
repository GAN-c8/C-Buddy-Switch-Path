# 遇卡换路原则：通用问题解决框架

## ⚡ 主动加载协议（v1.1.0 新增，最高优先级）

> **核心理念：加载是预防性的，不是补救性的。**
> 不要等到卡住再想起来，要在任务开始前就装好"安全网"。

### 必须主动加载的场景（满足任意一条即加载）

| 触发条件 | 典型例子 |
|----------|----------|
| 任务预计超过 **3个步骤** | Excel分析→图表→Word报告 |
| 涉及 **脚本执行** | Python / Node.js 脚本运行 |
| 涉及 **文件生成或批量操作** | 生成Word/Excel/PDF、批量重命名 |
| 涉及 **系统环境依赖** | 安装包、配置路径、调用Shell命令 |
| 涉及 **外部服务调用** | 发邮件、调API、访问网络 |
| 涉及 **桌面/文件夹整理** | 移动文件、建目录、清理旧版本 |
| 涉及 **自动化任务配置** | 创建/修改 WorkBuddy automation |

### 加载时机
```
接到任务
  ↓
判断：是否满足上表任意一条？
  ├─ 是 → 立即加载此 Skill（在第一步操作之前）
  └─ 否 → 可跳过，按普通流程执行

任务中途遇到第一次报错？
  → 立即补加载（亡羊补牢）
```

### 加载后的承诺
加载此 Skill 后，小C承诺：
- 同一路径失败 **≥2次** 时，**强制停止**，不再重试
- 换路前必须**说出根本原因**，不能无声更换
- 告知用户"开始遇卡换路，新方案是XXX"
- 记录失败路径，避免下次重蹈覆辙

## 概述

遇卡换路（Switch Path When Stuck）是一种系统化的问题解决框架，核心原则是：**当任务执行在同一方向上卡住超过2次时，立即停止当前方向，分析根本原因，选择一条完全不同的路径重新执行**。

此框架将"遇卡换路"从具体经验升级为可复用的通用能力，适用于：
- 技术障碍（代码、环境、依赖问题）
- 流程阻塞（工作流、自动化、系统集成）
- 创意困境（设计、文案、方案规划）
- 协作瓶颈（沟通、协调、资源分配）

## 工作流程决策树

```
遇到任务卡住
├─ 卡住次数 ≤ 1 → 继续尝试当前路径
└─ 卡住次数 ≥ 2 → 触发遇卡换路流程
    ├─ 步骤1：停 → 主动停止当前方向
    ├─ 步骤2：析 → 判断卡住的根本原因
    │   ├─ 环境问题（Python/Node版本、依赖包、系统配置）
    │   ├─ 方案错误（技术选型、架构设计、实现思路）
    │   ├─ 依赖缺失（外部API、数据源、权限配置）
    │   ├─ 资源限制（内存、存储、网络、时间）
    │   └─ 理解偏差（需求误解、目标错位、期望差异）
    │
    ├─ 步骤3：换 → 选择完全不同的路径
    │   ├─ 更换技术方案（换语言、换库、换架构）
    │   ├─ 更换执行环境（换工具、换平台、换方式）
    │   ├─ 更换数据来源（换API、换文件格式、换输入方式）
    │   ├─ 更换问题视角（换切入点、换优先级、换分解方式）
    │   └─ 更换协作方式（换沟通渠道、换资源分配、换责任人）
    │
    └─ 步骤4：再卡再换 → 新路径也卡住？重复1-3步
```

## 核心四步法

### 1. 停：主动停止当前方向

**触发条件**：同一任务在同一方向上失败**超过2次**
- 第一次失败：可能是偶然性错误，继续尝试
- 第二次失败：开始警惕，检查基本假设
- 第三次失败：**必须停止**，触发遇卡换路

**停止动作**：
- 立即放弃当前执行路径
- 保存当前状态和错误信息
- 明确告知用户"开始遇卡换路流程"

### 2. 析：判断卡住的根本原因

**根本原因分类矩阵**：

| 原因类型 | 典型表现 | 诊断方法 |
|----------|----------|----------|
| **环境问题** | "命令不存在"、"模块未找到"、"版本不兼容" | 检查`python --version`、`node -v`、`pip list` |
| **方案错误** | "逻辑死循环"、"性能极差"、"架构无法扩展" | 画出方案流程图，检查关键假设 |
| **依赖缺失** | "API返回404"、"数据库连接失败"、"权限拒绝" | 验证外部依赖是否可达、可用、有权限 |
| **资源限制** | "内存不足"、"磁盘空间满"、"超时" | 检查系统资源使用情况 |
| **理解偏差** | "用户说不是这个意思"、"需求变了"、"目标不对" | 重新与用户确认需求和期望 |

**诊断工具**：
- **环境诊断脚本**：`scripts/diagnose_environment.py`（快速检查系统状态）
- **方案审查清单**：`references/solution_review_checklist.md`
- **依赖验证工具**：`scripts/verify_dependencies.py`

### 3. 换：选择完全不同的路径

**路径切换策略库**：

#### 技术障碍切换策略
```
原路径：Python matplotlib 图表生成卡住
├─ 换路1：用Python自定义SVG函数绘图（避开matplotlib依赖）
├─ 换路2：用Node.js + Chart.js生成HTML图表（Web方案）
├─ 换路3：直接输出数据表格，让用户自行处理（简化方案）
└─ 换路4：改用第三方图表API（外部服务）
```

#### 环境问题切换策略
```
原路径：系统Python版本问题
├─ 换路1：使用独立Python环境（虚拟环境/容器）
├─ 换路2：切换到Node.js实现（换技术栈）
├─ 换路3：寻找无依赖的纯文本解决方案
└─ 换路4：手动安装正确版本并配置路径
```

#### 流程阻塞切换策略
```
原路径：自动化任务版本混乱
├─ 换路1：清理旧版本，建立唯一权威版本
├─ 换路2：改用更简单的定时脚本
├─ 换路3：人工干预+半自动化
└─ 换路4：重新设计整个自动化架构
```

### 4. 再卡再换：持续迭代

**核心原则**：
- 新路径也卡住？立即重复"停-析-换"流程
- 每次换路**必须换到完全不同的领域**
- 记录所有尝试路径，形成"避坑地图"
- 最多尝试**3条完全不同的路径**，然后寻求帮助

## 应用场景案例库

### 案例1：Excel数据分析 → Word报告（实际经验）
**问题**：Python matplotlib中文字体配置反复失败
**遇卡换路流程**：
1. **停**：第3次尝试`rcParams['font.sans-serif']`失败后停止
2. **析**：根本原因：matplotlib环境配置复杂，依赖系统字体包
3. **换**：
   - 换路1：`rcParams['font.family'] = 'Microsoft YaHei'`（直接使用系统字体）✅
   - 换路2：Python自定义SVG函数绘图（避开matplotlib）
   - 换路3：改用HTML+JavaScript图表（完全不同的技术方案）
4. **结果**：换路1成功，解决方案写入长期记忆

### 案例2：新闻简报自动化任务版本混乱
**问题**：早上8点执行了错误的旧脚本
**遇卡换路流程**：
1. **停**：发现格式不符合要求，立即停止分析原因
2. **析**：根本原因：多个脚本并存，自动化任务指向错误版本
3. **换**：
   - 换路1：创建唯一权威脚本版本`daily-news-content.js`
   - 换路2：标记旧脚本为已弃用
   - 换路3：新建自动化任务指向正确脚本
   - 换路4：建立版本管理规范
4. **结果**：系统加固，永久解决版本混乱问题

### 案例4：AI工具本身反复遗忘加载Skill（2026-03-27 新增）
**问题**：小C在执行长任务时反复忘记主动加载遇卡换路Skill，灿哥多次提醒
**遇卡换路流程**：
1. **停**：意识到"靠被动提醒"这条路行不通，停止依赖运行时记忆
2. **析**：根本原因：Skill触发是被动的（等卡住才想起），没有强制预检机制；加载时机不明确
3. **换**：
   - 换路1：在MEMORY.md写入强制规则——长任务开始前必须加载（靠持久记忆约束）
   - 换路2：在SKILL.md头部写明主动触发场景清单（靠Skill自身描述约束）
   - 换路3：两者组合，双重保险（最终方案）✅
4. **结果**：v1.1.0 升级落地，主动加载协议正式写入Skill规范

> **教训**：框架本身也可能成为被遗忘的对象。解决方案不是"更努力地记"，而是"把规则写进系统"。

## 决策支持工具

### 1. 根本原因快速诊断表

当遇到问题时，按顺序检查：
```
[ ] 环境问题：运行环境是否配置正确？（版本、路径、依赖）
[ ] 方案错误：技术方案是否可行？（逻辑、性能、扩展性）
[ ] 依赖缺失：外部依赖是否可用？（API、数据库、文件）
[ ] 资源限制：系统资源是否充足？（内存、磁盘、网络）
[ ] 理解偏差：需求理解是否正确？（目标、范围、期望）
```

### 2. 换路策略选择矩阵

| 原问题类型 | 推荐换路方向 | 具体案例 |
|------------|--------------|----------|
| 技术依赖问题 | 换技术栈/换库 | matplotlib → 自定义SVG |
| 环境配置问题 | 换环境/换方式 | 系统Python → 独立环境 |
| 流程混乱问题 | 换架构/换规范 | 多版本并存 → 唯一权威版本 |
| 外部依赖问题 | 换来源/换方案 | API不可用 → 模拟数据 |
| 理解偏差问题 | 换沟通/换确认 | 重新确认需求 |

### 3. 遇卡换路执行清单

开始换路前必须完成：
```
✅ 停止当前执行（不再重试同一路径）
✅ 记录失败原因和尝试次数
✅ 分析根本原因（使用诊断表）
✅ 选择完全不同的新路径（参考策略矩阵）
✅ 告知用户"开始换路，新方案是XXX"
✅ 执行新路径，监控结果
✅ 新路径也卡住？重复流程
```

## 使用时机

> **v1.1.0 起：主动加载优先于被动触发。**

**必须主动加载（任务开始前）**：
- 任务满足"主动加载协议"中的任意一条触发条件
- 这是强制规则，不是建议

**遇卡后触发（任务中途）**：
- 同一任务在同一方向上失败超过2次
- 遇到技术障碍反复无法解决
- 环境配置问题困扰已久
- 流程阻塞影响工作效率

**不需要加载**：
- 单步骤、无依赖的简单操作（如读一个文件、回答问题）
- 明确知道解决方案，只需要执行时间
- 用户要求必须按特定方式完成，无法换路

## 技能优势

1. **避免死磕**：强制换路机制防止在错误方向上浪费时间
2. **系统化思考**：提供完整的"停-析-换"思考框架
3. **经验传承**：案例库和避坑地图积累组织知识
4. **适应性**：适用于技术、流程、协作等各类问题
5. **可扩展**：不断丰富案例库和换路策略
