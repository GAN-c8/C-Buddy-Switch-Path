# 遇卡换路案例研究

基于实际工作经验整理的遇卡换路应用案例，每个案例包含问题描述、换路过程和经验总结。

## 案例1：Excel数据分析 → Word报告生成

### 问题描述
**任务**: 读取Excel数据，分析后生成包含图表的Word报告  
**卡住点**: Python matplotlib中文字体显示乱码  
**尝试次数**: 3次在同一方向上失败

### 遇卡换路过程

#### 步骤1：停
- **第1次尝试**: `rcParams['font.sans-serif'] = ['SimHei', 'Microsoft YaHei']`
  - 结果：部分图表正常，部分仍乱码
- **第2次尝试**: 安装额外中文字体包
  - 结果：权限问题，安装失败
- **第3次尝试**: 修改系统字体路径配置
  - 结果：系统配置复杂，卡住超过30分钟
  
**停止决策**: 第3次失败后，立即停止当前方向，触发遇卡换路。

#### 步骤2：析
**根本原因分析**:
1. **环境问题**: matplotlib字体配置依赖系统字体和字体路径，环境复杂
2. **方案错误**: 试图通过配置解决所有场景，未考虑环境差异
3. **依赖缺失**: 没有统一的字体解决方案

**诊断工具使用**:
```bash
# 运行环境诊断脚本
python scripts/diagnose_environment.py
```
诊断结果：Python环境正常，但字体配置复杂。

#### 步骤3：换
**换路策略选择**:

| 路径 | 描述 | 投入 | 选择理由 |
|------|------|------|----------|
| **换路1** | `rcParams['font.family'] = 'Microsoft YaHei'` | ⭐⭐ | 直连系统字体，避开复杂配置 |
| 换路2 | Python自定义SVG函数绘图 | ⭐⭐⭐ | 完全避开matplotlib依赖 |
| 换路3 | 改用HTML+JavaScript图表 | ⭐⭐⭐⭐ | Web方案，跨平台兼容 |

**执行换路1**:
```python
# 原方案（复杂配置）
plt.rcParams['font.sans-serif'] = ['SimHei', 'Microsoft YaHei', 'Arial Unicode MS']
plt.rcParams['axes.unicode_minus'] = False

# 新方案（简单直连）
plt.rcParams['font.family'] = 'Microsoft YaHei'
```

#### 步骤4：结果
- **成功**: 换路1立即生效，所有图表中文字体正常显示
- **经验固化**: 
  1. 将解决方案写入长期记忆
  2. 建立"Excel→Word报告标准流程"
  3. 总结"遇卡换路原则"

### 经验总结
1. **识别卡住信号**: 同一方向失败3次是明确换路信号
2. **根本原因分析**: 区分环境问题 vs 方案问题
3. **换路策略**: 从简单方案开始尝试
4. **知识管理**: 成功经验立即固化到工作流程

## 案例2：新闻简报自动化任务版本混乱

### 问题描述
**任务**: 每天8点自动发送标准格式新闻简报  
**卡住点**: 早上执行了错误的旧脚本，格式不符合要求  
**尝试次数**: 发现问题时已执行错误版本

### 遇卡换路过程

#### 步骤1：停
- **发现问题**: 用户反馈格式不正确
- **立即停止**: 停止分析所有相关自动化任务
- **保存状态**: 记录当前所有脚本版本和执行日志

#### 步骤2：析
**根本原因分析**:
1. **版本管理缺失**: 多个新闻脚本文件并存
2. **自动化配置混乱**: 可能有多个自动化任务指向不同脚本
3. **缺乏标准化**: 没有唯一的权威脚本版本

**问题症状**:
- `send-news.js` (旧版，简易格式)
- `daily-news-content.js` (标准版，正确格式)
- 自动化任务可能指向了错误版本

#### 步骤3：换
**换路策略选择**:

| 路径 | 描述 | 投入 | 选择理由 |
|------|------|------|----------|
| **换路1** | 建立唯一权威版本 | ⭐⭐ | 解决根本问题，防止再次发生 |
| 换路2 | 清理旧版本，标记弃用 | ⭐ | 减少混乱，明确指导 |
| 换路3 | 重建自动化任务 | ⭐⭐ | 确保指向正确版本 |
| 换路4 | 建立版本管理规范 | ⭐⭐⭐ | 长期预防机制 |

**执行多路并行换路**:
1. **换路1**: 确认`daily-news-content.js`为唯一权威版本
2. **换路2**: 在`send-news.js`文件头添加"已弃用"注释
3. **换路3**: 创建新自动化任务"每日新闻简报（标准格式）"
4. **换路4**: 建立脚本命名规范和版本管理流程

#### 步骤4：结果
- **立即修复**: 手动执行标准格式脚本，发送正确简报
- **系统加固**: 建立版本管理，防止再次混乱
- **长期预防**: 将经验写入系统稳定性优化规范

### 经验总结
1. **预防优于修复**: 版本混乱是系统性问题，需要架构层面解决
2. **多路并行**: 复杂问题可能需要多个换路策略同时执行
3. **标准化**: 建立唯一权威版本是防止混乱的关键
4. **透明化**: 明确标记弃用版本，避免误用

## 案例3：外部API调用失败

### 问题描述
**任务**: 调用天气API获取数据并展示  
**卡住点**: API返回403 Forbidden错误  
**尝试次数**: 2次调用失败

### 遇卡换路过程

#### 步骤1：停
- **第1次尝试**: 直接调用API，返回403
- **第2次尝试**: 检查API密钥，重新调用，仍返回403
- **停止决策**: 2次失败后停止，分析原因

#### 步骤2：析
**根本原因分析**:
1. **外部依赖问题**: API密钥可能过期或权限不足
2. **方案风险**: 强依赖外部服务，无降级方案
3. **理解偏差**: 可能API使用方式错误

**诊断检查**:
- API文档：需要每小时刷新token
- 当前token：已超过24小时未刷新
- 错误代码：403表示认证失败

#### 步骤3：换
**换路策略选择**:

| 路径 | 描述 | 投入 | 选择理由 |
|------|------|------|----------|
| **换路1** | 刷新API令牌 | ⭐ | 最简单直接的解决方案 |
| 换路2 | 使用模拟数据 | ⭐⭐ | 保证功能可用，不依赖API |
| 换路3 | 寻找替代API | ⭐⭐⭐ | 长期解决方案，降低依赖风险 |
| 换路4 | 改为手动输入 | ⭐ | 兜底方案，保证任务完成 |

**执行换路1+换路2组合**:
1. **换路1**: 实现API令牌自动刷新机制
2. **换路2**: 实现模拟数据生成，API失败时自动降级
3. **组合优势**: 既尝试修复问题，又有兜底方案

**实现代码**:
```python
def get_weather_data():
    try:
        # 尝试真实API
        data = call_real_api()
        return data
    except APIError:
        # API失败时使用模拟数据
        return generate_mock_data()
```

#### 步骤4：结果
- **立即解决**: 刷新令牌后API调用成功
- **系统加固**: 实现自动刷新和降级机制
- **风险降低**: 不再强依赖单一外部服务

### 经验总结
1. **外部依赖风险**: 外部服务可能随时失效，必须有降级方案
2. **组合换路**: 可以同时尝试多个换路策略
3. **兜底机制**: 关键功能必须有备用方案
4. **自动化处理**: 常见问题（如令牌刷新）应该自动化

## 案例4：大型数据处理内存溢出

### 问题描述
**任务**: 处理10GB的CSV文件进行分析  
**卡住点**: 内存不足，程序崩溃  
**尝试次数**: 2次尝试读取整个文件到内存

### 遇卡换路过程

#### 步骤1：停
- **第1次尝试**: `pd.read_csv('10gb.csv')` → 内存溢出
- **第2次尝试**: 增加内存限制参数 → 仍失败
- **停止决策**: 2次失败后停止，重新设计方案

#### 步骤2：析
**根本原因分析**:
1. **资源限制**: 物理内存不足处理10GB文件
2. **方案错误**: 试图一次性加载整个文件
3. **技术选型**: Pandas不适合超大文件处理

**资源诊断**:
- 可用内存：8GB
- 文件大小：10GB
- 数据特点：按时间顺序排列，可分片处理

#### 步骤3：换
**换路策略选择**:

| 路径 | 描述 | 投入 | 选择理由 |
|------|------|------|----------|
| **换路1** | 分批读取处理 | ⭐⭐ | 最直接，利用现有技术栈 |
| 换路2 | 使用数据库 | ⭐⭐⭐ | 专业数据管理，但增加复杂度 |
| 换路3 | 使用Dask分布式 | ⭐⭐⭐⭐ | 适合大数据，但学习成本高 |
| 换路4 | 抽样分析 | ⭐ | 快速验证，精度可能不足 |

**执行换路1**:
```python
# 原方案（一次性加载）
df = pd.read_csv('10gb.csv')

# 新方案（分批处理）
chunk_size = 100000  # 每次处理10万行
results = []

for chunk in pd.read_csv('10gb.csv', chunksize=chunk_size):
    # 处理每个分片
    processed = process_chunk(chunk)
    results.append(processed)

# 合并结果
final_result = combine_results(results)
```

#### 步骤4：结果
- **成功解决**: 分批处理避免内存溢出
- **性能可接受**: 处理时间增加但稳定运行
- **扩展性**: 支持任意大小的文件

### 经验总结
1. **资源感知**: 处理前评估数据大小和可用资源
2. **渐进处理**: 大数据必须分而治之
3. **技术匹配**: 选择适合数据规模的技术方案
4. **监控调整**: 根据实际情况动态调整分片大小

## 通用换路模式总结

### 模式1：环境问题 → 更换环境/工具
- **典型表现**: 配置复杂、版本冲突、权限问题
- **换路方向**: 独立环境、不同工具、云服务
- **成功案例**: matplotlib字体问题 → 直连系统字体

### 模式2：方案错误 → 更换实现方案
- **典型表现**: 性能差、复杂度高、不可扩展
- **换路方向**: 不同算法、简化设计、成熟框架
- **成功案例**: 一次性加载大文件 → 分批处理

### 模式3：外部依赖 → 更换来源/降级
- **典型表现**: API失败、服务不可用、数据缺失
- **换路方向**: 备用API、模拟数据、手动输入
- **成功案例**: 天气API失败 → 令牌刷新+模拟数据

### 模式4：资源限制 → 优化/分批/外部化
- **典型表现**: 内存不足、超时、存储空间满
- **换路方向**: 资源优化、分批处理、云资源
- **成功案例**: 10GB文件处理 → 分批读取

### 模式5：理解偏差 → 重新沟通/验证
- **典型表现**: 需求不符、期望错位、频繁变更
- **换路方向**: 原型验证、分阶段确认、重新沟通
- **成功案例**: 功能不符 → 快速原型确认

## 换路策略选择指南

### 优先级顺序
1. **最小投入路径**: 先尝试最简单的换路方案
2. **根本解决路径**: 针对根本原因的系统性解决方案
3. **临时绕过路径**: 快速解决问题，可能不是最优但可用
4. **长期优化路径**: 需要较大投入但长期受益

### 决策矩阵
| 问题严重性 | 时间紧迫性 | 推荐换路策略 |
|------------|------------|--------------|
| 高 | 高 | 临时绕过路径（快速解决） |
| 高 | 低 | 根本解决路径（系统修复） |
| 低 | 高 | 最小投入路径（简单尝试） |
| 低 | 低 | 长期优化路径（全面改进） |

### 风险评估
- **低风险换路**: 尝试新方案，原方案可随时恢复
- **中风险换路**: 需要一定投入，但失败影响可控
- **高风险换路**: 重大架构变更，失败可能影响项目
