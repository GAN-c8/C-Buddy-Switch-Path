# 方案审查清单

在"遇卡换路"流程的"析"步骤中，使用此清单审查技术方案是否合理。

## 一、基础可行性检查

### 1.1 技术选型审查
```
[ ] 是否选择了合适的技术栈？（Python/Node.js/其他）
[ ] 技术栈是否成熟稳定？（社区支持、文档完善）
[ ] 是否有更简单的替代方案？
[ ] 团队成员是否熟悉此技术？
[ ] 是否符合项目长期技术路线？
```

### 1.2 依赖审查
```
[ ] 依赖的第三方库是否活跃维护？
[ ] 依赖数量是否可控？（避免依赖地狱）
[ ] 是否有版本冲突风险？
[ ] 是否需要特定环境配置？
[ ] 是否有许可证合规问题？
```

### 1.3 环境兼容性
```
[ ] 是否兼容目标操作系统？（Windows/macOS/Linux）
[ ] 是否兼容Python/Node版本要求？
[ ] 是否需要管理员权限？
[ ] 是否需要网络连接？
[ ] 是否需要特定硬件资源？
```

## 六、审查结果处理

### 6.1 通过标准
- ✅ 所有"基础可行性"检查通过
- ✅ 80%以上"架构设计"检查通过  
- ✅ 关键"实现细节"检查通过
- ✅ 无高风险项目

### 6.2 需要优化
- ⚠️  1-3项"基础可行性"未通过
- ⚠️  30%-50%"架构设计"未通过
- ⚠️  存在中等风险项目

### 6.3 建议换路
- ❌ 超过3项"基础可行性"未通过
- ❌ 超过50%"架构设计"未通过
- ❌ 存在高风险项目
- ❌ 明显超出当前技术能力

## 七、换路决策指南

根据审查结果决定是否需要换路：

### 7.1 继续当前路径（条件）
```
✓ 技术方案整体可行
✓ 风险可控
✓ 有明确解决方案
✓ 时间预算充足
✓ 团队能力匹配
```

### 7.2 考虑换路（预警信号）
```
⚠️  关键依赖无法解决
⚠️  性能问题无法优化
⚠️  架构存在重大缺陷
⚠️  超出团队技术能力
⚠️  时间预算严重不足
```

### 7.3 必须换路（危险信号）
```
❌ 基础环境无法配置
❌ 核心功能无法实现
❌ 安全漏洞无法修复
❌ 性能完全不可接受
❌ 用户需求无法满足
```

## 八、常见问题模式及换路建议

### 8.1 环境配置反复失败
**模式**: `pip install`失败、版本冲突、权限问题反复出现
**换路建议**:
1. 使用独立虚拟环境（venv/conda）
2. 换用Docker容器
3. 切换到不同技术栈
4. 使用云开发环境

### 8.2 第三方API不稳定
**模式**: API频繁超时、返回错误、服务中断
**换路建议**:
1. 实现本地缓存
2. 寻找替代API
3. 使用模拟数据
4. 改为离线处理

### 8.3 性能问题无法解决  
**模式**: 处理速度慢、内存占用高、CPU使用率高
**换路建议**:
1. 更换算法（如递归→迭代）
2. 分批处理大数据
3. 使用更高效的数据结构
4. 引入异步处理

### 8.4 架构复杂度过高
**模式**: 代码复杂度爆炸、维护困难、bug频发
**换路建议**:
1. 简化架构设计
2. 拆分模块降低耦合
3. 使用成熟框架
4. 重新评估需求范围

---

**使用提示**:
1. 在"遇卡换路"的"析"步骤中使用此清单
2. 对每个检查项客观评分（✓/⚠️/❌）
3. 根据审查结果决定是否换路
4. 记录审查过程，形成经验积累
